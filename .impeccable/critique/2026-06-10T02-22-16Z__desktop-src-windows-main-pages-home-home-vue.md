#### Anti-Patterns Verdict / 反模式判断

**LLM 评估：** 中等 AI 味风险。Home 页具备真实产品结构，但视觉修辞滑向了生成式“高级仪表盘”套路：玻璃拟态侧栏、发光激活条、3D 环境装饰、动画指标 chip、点阵空状态，以及更像 SaaS 证明材料而非冷静桌面工作台的头部统计。

**确定性扫描：** 干净。`detect.mjs --json desktop/src/windows/main/pages/Home/Home.vue` 返回 `[]`，退出码 0。没有规则命中、文件位置或误报。

**视觉叠层：** 常规路径下没有可靠的用户可见叠层。独立 Vite 浏览器加载缺少 Electron preload API 会失败，`desktop/index.html` 的 CSP 也会阻止 `127.0.0.1:8400` 的正常 detector 注入。Assessment B 证明该路由可通过临时 preload stub 渲染，也能通过 DevTools evaluation 运行 `detect.js`，但这不等同于完整 Electron 运行时里的用户可见叠层证据。

#### Overall Impression / 整体印象

Home 页的骨架是对的：左侧分类 rail + 中央可配置工作台，非常适合 GuYanTools 的个人工具枢纽定位。最大机会是让“创建工作台”的路径变得可见、可靠、可恢复。现在页面更强调壳层的视觉表现，却把真正让壳层有用的动作藏得太深。

#### What's Working / 做得好的地方

- 左侧分类 rail 和中央网格适合作为个人桌面工具工作台。
- 实现上大体使用共享 Vue 组件和语义 token，没有完全脱离设计系统。
- 个性化被当作一等产品能力处理：支持主题感知背景和接近真实尺寸的预览。

#### Priority Issues / 优先问题

**[P1] 主操作隐藏在右键菜单中**

为什么重要：新用户看到“暂无首页分类”/“从左侧添加一个类别”，但真实添加入口藏在侧栏右键菜单里。新建组件、区域个性化、顶栏/侧栏个性化、编辑和删除也存在同样问题。

修复：增加可见 affordance：空状态里的“添加类别”按钮、侧栏中的持久轻量添加控件、工作区里的“新建组件”按钮，以及选中小组件后的可见操作入口。保留右键作为加速器，而不是唯一入口。

建议命令：`/impeccable onboard desktop/src/windows/main/pages/Home/Home.vue`

**[P1] 破坏性操作缺少可见恢复路径**

为什么重要：个性化工作台里的误删成本很高。当前小组件删除通过右键菜单触发，看起来会立即删除。

修复：增加确认或撤销 toast，明确显示被删除的小组件名称并提供恢复。可以延迟持久化到撤销窗口结束，或保留短期恢复快照。

建议命令：`/impeccable harden desktop/src/windows/main/components/CompArea/GridItem.vue`

**[P1] 可访问路径不完整**

为什么重要：小组件是带双击/右键行为的 `div`，分类激活状态没有清晰 ARIA 表达，隐藏菜单让键盘用户和屏幕阅读器用户很难完成主要流程。

修复：让 grid item 成为可聚焦控件，或暴露内部按钮；为打开、编辑、删除添加键盘等价路径；分类按钮使用 `aria-selected` 或 `aria-current`；确保菜单和组件创建流程不用鼠标也能抵达。

建议命令：`/impeccable audit desktop/src/windows/main/pages/Home/Home.vue`

**[P2] 头部对任务型界面来说过度设计**

为什么重要：3D 环境装饰、指标 chip、发光分割线、极小 uppercase 标签和入场动画把注意力从工作台转移到装饰层。

修复：把装饰性指标替换成操作性状态：当前配置文件、上次保存、布局健康、同步/持久化状态或快捷操作。删除默认 3D 头部装饰，或将其变成用户个性化预览，而不是默认 chrome。

建议命令：`/impeccable quieter desktop/src/windows/main/pages/Home/Home.vue`

**[P2] 个性化可能破坏可读性**

为什么重要：用户可以设置图片、视频、透明度和文字颜色。代码支持文字颜色变量，但 UX 需要护栏，防止个性化破坏对比度或造成动效疲劳。

修复：在个性化配置中加入对比度检查/警告，为图片和视频背景提供默认遮罩，并对背景视频、3D、入场动画添加 reduced-motion 处理。

建议命令：`/impeccable harden desktop/src/windows/main/components/ui/UiPersonalizationConfig.vue`

#### Persona Red Flags / 用户画像风险

**Alex（高效专家用户）：** 右键、拖拽和双击对他有效，但没有明显的命令式路径或快捷键层。立即删除且无撤销会降低他管理大量自定义组件时的信任。滚轮切换分类也可能在工作区滚动时造成意外。

**Sam（依赖无障碍的用户）：** 小组件卡片不是语义按钮，很多动作优先依赖鼠标/右键菜单，侧栏依赖极小标签和视觉激活样式。背景个性化和弱化文字 token 会带来对比度风险。

**Jordan（首次使用者）：** 空状态告诉他添加类别，但不显示按钮。“组件”“区域个性化配置”“保活网页”都假设用户已理解产品概念。紧凑侧栏会截断标签，让图标承担过多含义。

#### Minor Observations / 次要观察

- `home.scss` 的头部使用了 `clamp()` 和负 letter spacing，和产品设计文档中的固定 rem 字号规则冲突。
- 激活分类使用 3px 左侧强调条，违反了 no side-stripe 指导；可以改为完整激活表面、图标状态或选择环。
- 侧栏滚动按钮默认 opacity 为 0，滚动能力很弱。
- CSS 注释和命名大量出现 “Premium”“Glassmorphism”“3D ambient”，和过度设计风险相互印证。
- 错误文案应该告诉用户下一步做什么，而不是让用户“检查布局数据”。

#### Questions to Consider / 值得思考的问题

- 如果 Home 的第一可见动作是“添加你的第一个工具”，而不是统计数字，会是什么样？
- 个性化是产品能力的核心，还是现在正在和用户的真实工作争夺注意力？
- 如果右键不存在，首次使用者还能在 60 秒内搭建一个可用工作台吗？
