# 桌面端插件系统现状分析

## 结论

当前桌面端并没有实现一个“真正可用的插件系统”。

现有实现更准确地说，是一个 **UI 插件加载雏形**：

- 可以在主进程初始化时准备插件目录和本地插件配置。
- 可以用 `npm install/link/uninstall/update` 的方式管理外部包。
- 可以把插件包里的入口页面作为 `file://` 页面加载到一个独立的 `WebContentsView` 中。
- 可以给插件页注入一个非常薄的 `pluginAPI` 预加载对象。

这里先纠正一个边界：

- 本文档本身以 Markdown 形式持久化到仓库里。
- 这不等于“插件的运行结果必须持久化到 Markdown”。

本文讨论的插件定义更准确地说应当是：

> 使用应用提供的能力/接口，实现新的页面或系统级功能。

按这个定义，当前实现距离目标仍然很远。它还没有形成：

- 明确的宿主能力接口层
- 页面/路由/入口注册机制
- 系统级功能挂载点
- 插件生命周期和权限模型
- 可运营的安装、启停、升级、卸载、错误恢复与兼容性体系

## 当前设计实际上是什么

从代码上看，当前插件设计的核心路径是：

1. 应用启动时加载 `plugin_core/index.ts`
2. `local_plugin.ts` 在用户数据目录下准备 `guyan-plugins/` 和 `guyan-local-plugin.json`
3. `PluginManager` 负责在该目录里执行 npm 命令，安装/更新/卸载插件包
4. 宿主通过 `getPluginWebContentsView()` 把插件包里的 `main` 文件作为本地页面载入
5. 插件页面拿到一个 `pluginAPI`，但这个 API 目前几乎没有实际能力

这说明现阶段的设计中心不是“能力扩展”，而是“把一个外部网页/前端页嵌入到 Electron 里运行”。

## 当前已经实现了什么

### 1. 插件安装目录与本地配置初始化

`desktop/src/core/plugin_core/local_plugin.ts` 做了几件基础工作：

- 以 `app.getPath('userData')/guyan-plugins` 作为插件安装目录
- 创建 `guyan-local-plugin.json` 作为本地插件配置文件
- 把配置文件内容读入 `global.LOCAL_PLUGINS.plguins`

这解决的是“插件文件放哪”和“本地配置从哪读”的问题。

### 2. 基于 npm 的插件包管理雏形

`desktop/src/core/plugin_core/plugin_manager.ts` 已经具备这些方法：

- `install()`
- `udpate()`
- `uninstall()`
- `list()`
- `upgradePlugin()`
- `getPluginInfo()`

本质上它是拿一个独立目录当作插件工作区，然后在里面执行 `npm install` / `npm link`。

这说明当前插件分发模型默认是“npm 包”。

### 3. 插件元信息模型雏形

`desktop/src/core/@types/plugin.d.ts` 定义了 `PluginInfo`：

- `type: "ui" | "system"`
- `name`
- `pluginName`
- `author`
- `description`
- `main`
- `version`
- `logo`

也就是说，作者已经有“区分 UI 插件和 system 插件”的意图，只是 system 方向还没有落地。

### 4. 独立插件预加载脚本

`desktop/forge.config.ts` 里已经把 `src/core/plugin_core/preload.plugin.ts` 作为单独 preload 构建目标打进 Electron 构建流程。

这说明宿主已经意识到：

- 插件页需要独立 preload
- 插件页和主窗口 preload 应该分离

这是一个正确方向。

### 5. 插件页面载入能力

`local_plugin.ts` 里的 `getPluginWebContentsView()` 已经能：

- 按插件清单解析入口文件
- 创建 `WebContentsView`
- 注入插件 preload
- 监听加载成功、失败和控制台日志
- 把插件页面通过 `file://` 方式载入

这属于“插件容器”最早期的一层。

### 6. 宿主应用已有一套可复用的数据能力，但还没开放给插件

桌面端现在已经有：

- `homeLayoutApi` 这套 IPC 桥
- Rust 核心库暴露的用户、项目、设置、首页布局数据库能力

说明宿主本身并不是完全没有能力层，而是 **这些能力目前只服务自家页面，没有变成插件可调用的稳定接口**。

## 现有雏形里还存在的直接实现问题

除了架构层面的缺口，当前这套雏形本身也还有一些明显未完工点：

- `downloadPlugin()` 是空实现。
- `importFromLocalFiles()` 是空实现。
- 主窗口里真正挂载插件 `WebContentsView` 的代码被注释掉了。
- `preload.plugin.ts` 返回的是写死的测试信息，不是实际插件上下文。
- `PluginManager.upgradePlugin()` 里的 registry URL 写成了 `https://registry.npmmirror.com/{pluginName}`，字面量没有插值。
- `PluginCache` 类型声明为 `number`，但缓存的其实是版本字符串。

也就是说，当前不仅“离真正可用的插件系统很远”，就连“作为一个稳定的 UI 插件试验框架”也还没有完全收口。

## 当前没有实现，或者只停留在概念上的部分

## 1. “system 插件”实际上没有实现

虽然 `PluginInfo.type` 里有 `"system"`，但：

- `desktop/src/core/plugin_core/register_system_plugins.ts` 是空文件
- `desktop/src/main/index.ts` 里有 `systemPlugins` 字段，但没有任何使用

所以 system 插件现在只是类型名，不是实际能力。

## 2. 插件并没有真正接入主应用界面

当前主窗口代码里，加载插件 `WebContentsView` 的逻辑是注释掉的。

这意味着：

- 应用正常运行路径里并不会真正显示插件
- 插件没有被接进首页、侧边栏、路由、标签页或独立窗口体系

从产品层面看，这还是实验代码，不是正在工作的插件框架。

## 3. 没有页面注册机制

如果插件要“实现新的页面”，至少需要一种注册方式，例如：

- 注册路由
- 注册首页卡片
- 注册侧边栏菜单
- 注册设置页子页
- 注册独立窗口/弹层

但当前桌面路由只有：

- `/home`
- `/settings`

没有动态路由，没有插件路由装配，没有页面挂载协议。

## 4. 没有系统级能力扩展点

真正的系统级插件至少要能挂进这些宿主能力之一：

- 菜单命令
- 快捷键
- 文件导入导出
- 后台任务
- 项目工作流
- 数据转换管线
- 事件订阅

当前代码里没有看到任何插件级命令总线、事件总线、后台任务注册、菜单注册、快捷键注册接口。

## 5. `pluginAPI` 目前基本不可用

`desktop/src/core/plugin_core/preload.plugin.ts` 现在只暴露了：

- `getPluginInfo()`，而且返回的是写死的假数据
- `send(message)`，本质上只是 `console.log`

这意味着插件拿不到：

- 当前工作区信息
- 当前项目信息
- 用户设置
- 应用数据库能力
- 文件系统能力
- 页面注册能力
- 命令执行能力
- 事件订阅能力

也就是说，插件并不能“基于宿主接口开发功能”。

## 6. 没有插件数据与宿主资源访问模型

既然插件目标是“基于宿主能力实现新页面/系统功能”，那插件至少需要一套受控的数据访问模型，例如：

- 访问当前工作区、项目、设置、用户信息
- 调用宿主已有数据库能力
- 读写插件自己的配置和状态
- 在必要时申请宿主级资源访问能力

当前仓库里没有看到面向插件的统一资源访问抽象，只有宿主页面自己在用的 API。

所以插件现在无法稳定地复用应用已有能力，只能作为一个被嵌入的外部页面存在。

## 7. 没有插件权限模型

如果插件未来要调用文件系统、数据库、项目、窗口能力，就必须有权限边界。

但当前插件容器配置是：

- `nodeIntegration: true`
- `webSecurity: false`
- `webviewTag: true`

这不是“插件通过宿主 API 受控扩展”，而是“插件页直接拥有非常强的运行权限”。

这会带来两个问题：

1. 安全上过于宽松，第三方插件几乎等价于本机代码执行。
2. 架构上绕开了宿主 API，插件作者没有动力走“标准能力接口”。

如果目标是真正可用的插件系统，这个方向需要收紧，而不是继续放开。

## 8. 没有插件生命周期管理

现在没有这些基本生命周期能力：

- 安装后注册
- 启动时发现
- 启用/禁用
- 升级迁移
- 卸载清理
- 崩溃恢复
- 版本兼容校验

`PluginManager` 虽然有安装命令，但没有形成完整的插件状态管理闭环。

## 9. 没有稳定的插件 Manifest 规范

当前 `PluginInfo` 只描述了很少的字段，缺失关键元数据，例如：

- 最低宿主版本
- 插件 API 版本
- 权限声明
- 提供的页面/命令/部件
- 激活条件
- 配置 schema
- 数据目录要求

没有 manifest 规范，就很难做兼容性、审核、安装校验和运行时装配。

## 10. 没有面向插件的错误隔离与观测能力

现在最多只是监听了：

- `did-finish-load`
- `did-fail-load`
- `console-message`

但真正可用的插件系统还需要：

- 插件运行状态面板
- 错误上报
- 超时控制
- 资源占用监控
- 崩溃后的禁用/回退
- 宿主级错误边界

这些都还没有。

## 11. 没有插件管理 UI

一个可用的插件系统通常要有：

- 插件市场或本地导入入口
- 已安装插件列表
- 启用/禁用开关
- 升级按钮
- 权限说明
- 配置页面
- 日志入口

现在这些 UI 也不存在。

## 12. 首页“组件”系统和插件系统还没打通

首页当前已经实现了分类、卡片布局、持久化和拖拽，但它本质上还是宿主自带工作台。

更关键的是：

- `GridItem.action` 只是数据字段
- `CompArea.vue` 里的 `handleItemOpen()` 还是 `TODO`

这说明首页卡片甚至还没有一个稳定的“点击后执行什么”的宿主执行模型，自然也还谈不上把插件卡片挂进去运行。

## 对“真正可用插件系统”的差距判断

如果按成熟度分层，当前大概只完成了：

### 已完成

- 插件目录初始化
- npm 包安装思路
- 独立 preload 构建
- 插件页面容器雏形

### 部分完成

- 插件元信息模型
- 插件加载实验路径

### 基本未完成

- 宿主能力 API
- 动态页面注册
- 系统级扩展点
- 插件数据与资源访问模型
- 生命周期管理
- 权限与安全
- 插件管理 UI
- 测试与兼容性策略

所以严格讲，当前实现离“真正可用”至少还差一个完整的宿主扩展层，而不是差几个收尾功能。

## 如果目标是“插件复用宿主能力扩展功能”，最少还需要补什么

如果不追求一步到位，先做一个可用版本，最少建议补这 6 层：

### 1. 定义插件 Manifest

至少补齐：

- `id`
- `name`
- `version`
- `hostApiVersion`
- `type`
- `entry`
- `permissions`
- `contributes.pages`
- `contributes.widgets`
- `contributes.commands`

### 2. 定义宿主插件 API，而不是让插件直接拿 Node 权限

建议把插件能做的事收敛成白名单 API，例如：

- `pluginAPI.workspace.getCurrentProject()`
- `pluginAPI.pages.open(id, params)`
- `pluginAPI.widgets.register(...)`
- `pluginAPI.commands.execute(id, payload)`
- `pluginAPI.storage.get/set(...)`

### 3. 增加宿主资源访问与持久化服务

建议由宿主主进程实现，而不是让插件自己直连 Node 或数据库：

- 统一插件配置存储
- 统一插件状态持久化
- 统一宿主资源访问入口
- 统一错误处理与审计

这样插件只关心“实现什么功能”，不关心“底层资源怎么安全访问”。

### 4. 增加页面和入口贡献点

至少支持两类：

- 新页面：可挂到路由或独立容器
- 新入口：可挂到首页卡片、侧边栏、菜单、命令面板

否则插件即使能运行，也无法被用户触达。

### 5. 增加权限与隔离

建议至少做到：

- 默认关闭 `nodeIntegration`
- 保持 `contextIsolation`
- 只通过 preload 暴露白名单 API
- 权限按 manifest 声明
- 安装或首次启用时展示权限说明

### 6. 增加插件状态管理

至少需要：

- 安装记录
- 启用/禁用状态
- 宿主版本兼容校验
- 插件错误状态
- 升级迁移钩子

## 更准确的现状判断

如果用一句话概括当前桌面端插件系统：

> 现在实现的是“外部前端页面装载器 + npm 包管理雏形”，还不是“基于宿主能力扩展页面与系统功能的插件平台”。

而你定义的那种插件系统，核心不在于“能不能把 HTML 跑起来”，而在于：

- 宿主提供了哪些稳定能力
- 插件如何声明自己需要什么能力
- 插件如何把能力挂进产品入口
- 插件如何安全、稳定地访问宿主资源

这四件事，当前基本都还没真正开始实现。

## 建议的下一步实现顺序

如果后续要继续推进，我建议按下面顺序做，而不是先继续堆 `WebContentsView`：

1. 先定义插件 Manifest 和宿主 API 边界。
2. 先做宿主资源访问与插件状态持久化服务。
3. 再做页面/卡片/命令贡献点。
4. 再做启用、禁用、安装、升级管理。
5. 最后再补插件市场、本地导入和更复杂的运行容器。

否则很容易继续得到一个“能跑 demo 页面，但不能稳定产出业务能力”的插件壳。
