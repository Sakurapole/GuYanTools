# Product

## Register

product

## Users

GuYanTools 面向偏技术、偏效率的桌面用户：他们希望拥有一个个人工作台，而不是在一堆彼此割裂的小工具之间来回切换。典型用户会在同一个持久化工作区里处理终端与 SSH 会话、FTP 和文件传输、Todo 计划、知识整理、AI 问答与研究、内置网页工具、插件和速记内容。

主要使用场景是长时间、专注的桌面工作：重复的日常操作、密集配置、多任务切换、本地优先流程，以及对可靠性、可恢复状态、键盘友好操作和高信息密度的要求。这里的设计目标不是营销式惊艳，而是让用户长时间使用也保持清醒、可控、可信。

## Product Purpose

GuYanTools 的目标是把实用桌面能力统一到一个可配置的工具枢纽里。成功的体验应当让用户快速组织自己的工作台、启动或恢复核心工具、信任长时间运行的任务，并在不牺牲清晰度和一致性的前提下完成视觉个性化。

这个产品应该像一个可靠的本地控制台：足够安静，适合长时间工作；足够精确，支撑开发者工作流；足够灵活，容纳个人组织方式。

## Brand Personality

精确、冷静、可扩展。

当前文档化的视觉方向是 “Lucid Architect”：清晰、轻透、以青蓝色为重点的界面，把应用当作一件精密的生产力仪器。文案语气应直接、友好、可操作；现有 UI 已大量使用中文业务文案，后续应继续中文优先。避免空泛宏大，状态和错误都要用普通用户能理解的方式解释。

## Anti-references

不要把应用界面做成 SaaS 落地页。避免超大 hero 修辞、重复的指标卡模板、泛 AI 工具紫色渐变、无意义玻璃拟态，以及不能传达状态的装饰性动效。

避免不一致的一次性控件、本地硬编码颜色体系、随意的 z-index、业务页面私有阴影/圆角体系、默认用模态框解决问题、被滚动容器裁切的下拉菜单，以及为了“有风格”而重新发明标准桌面交互。

避免厚重不透明的仪表盘盒子和无谓分割线。现有设计系统已经提供了语义 token、可复用 UI 组件、色调层级和 body 级固定浮层；新增界面应优先沿用这些能力。

## Design Principles

1. 任务流畅优先于视觉表演：每个页面都要减少它所负责工作流的阻力。
2. 一套组件语言：共享 UI 组件和 `--ui-*` 语义 token 是默认表达方式。
3. 高密度但可呼吸：复杂工具可以密集，但控件、状态和信息层级不能拥挤。
4. 个性化必须有护栏：背景、主题、小组件可以变化，但文字对比、焦点状态和布局稳定性不能被破坏。
5. 状态要可见且可恢复：加载、空状态、错误、选中、激活、流式输出、同步、禁用都要清楚表达。

## Accessibility & Inclusion

文字和控件目标应满足 WCAG AA 对比度。键盘焦点要始终可见，状态应通过语义、文案和结构表达，而不是只靠颜色。中文和英文文案都要在用户配置的基础字号下保持可读。

动效应短促、服务状态变化，并尊重 reduced-motion 预期。尤其在终端、传输、Todo、AI 和通知界面中，颜色不能成为唯一的状态提示。
